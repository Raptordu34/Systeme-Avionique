IMA — IOM, ADCN & Futur

Synthèse

  • IOM (I/O Module) — module d'entrées/sorties qui traduit les paquets ARINC 429 externes en trames AFDX internes (et inversement) ; composé d'une UE 429, d'une UR 429, d'une UTC et d'une UER AFDX
  • GWM (Gateway Module) — passerelle vers le réseau Ethernet externe ; même architecture UTC + UER AFDX que l'IOM, mais avec une UER Ethernet en lieu et place des unités 429
  • ADCN (Aircraft Data Communication Network) — réseau global de l'avion : AFDX comme colonne vertébrale, entouré de bus hétérogènes (ARINC 429, CAN, signaux analogiques et discrets)
  • SCI (Secure Communication Interface) — unique point de passage sécurisé entre le monde avionique certifié (AFDX) et le monde ouvert (Ethernet airline/Airbus) ; assure la translation d'adresses IP
  • IMA2G — seconde génération IMA : matériel distribué (DME), virtualisation PikeOS SSV, allocation dynamique des ressources ; projet européen SCARLETT (40 partenaires)

Le module IOM — I/O Module

Définition : L'IOM (I/O Module) est le module d'entrées/sorties de l'étagère IMA. Son rôle est d'assurer l'interface entre les équipements avioniques legacy connectés via bus ARINC 429 et le réseau AFDX interne à l'étagère. C'est le traducteur universel entre l'ancienne génération de bus et le nouveau réseau commuté.

Analogie : imaginez l'IOM comme un interprète bilingue en temps réel. D'un côté, il parle le « vieux dialecte » ARINC 429 aux équipements extérieurs (capteurs, actionneurs hérités d'anciennes architectures). De l'autre, il communique en AFDX avec les modules cœurs de l'étagère. Il traduit dans les deux sens simultanément, sans jamais introduire d'ambiguïté entre les deux mondes.

Architecture interne de l'IOM

Le module IOM empile quatre blocs fonctionnels du haut vers le bas :

  • UR 429 / UE 429 — Unités de Réception et d'Émission ARINC 429 : interfaces physiques vers les bus 429 extérieurs
  • Interface 429 — couche d'adaptation entre les unités 429 et l'UTC ; gère le formatage des paquets 429
  • UTC (Unité de Traitement des Communications) — cœur logique de l'IOM : orchestre la traduction bidirectionnelle 429 ↔ AFDX
  • UER AFDX (Unité d'Extrémité de Réseau AFDX) — point de connexion au réseau AFDX de l'étagère, commun à tous les modules IMA

UE 429 — Unité d'Émission ARINC 429

Le bus ARINC 429 est un bus mono-émetteur à diffusion : un seul équipement émet sur le bus, tous les récepteurs écoutent. L'identificateur de ligne est encapsulé dans chaque message, permettant à chaque récepteur de filtrer les messages qui le concernent.

Structure d'un paquet 429 : Un paquet 429 regroupe plusieurs labels 429. La taille d'un paquet dans la file FIFO est égale au nombre de labels qu'il contient. Quand la FIFO arrive à saturation, deux stratégies de gestion sont possibles : refus du paquet entrant (le paquet est rejeté en totalité), ou troncature (le paquet est retaillé pour tenir dans l'espace restant). L'UE 429 modélise le médium de communication comme une file FIFO dont un serveur consomme les paquets un par un avant de les émettre sur le bus.

UR 429 — Unité de Réception ARINC 429

La réception repose sur un mécanisme de filtrage appelé PAL (Piège À Label) : chaque UR 429 est configurée pour reconnaître un ensemble précis d'identificateurs de ligne. Tout message dont l'identifiant ne figure pas dans la liste de filtrage est silencieusement ignoré.

Dimensionnement du délai de reconnaissance : Le délai de reconnaissance d'un paquet par la UR 429 est soigneusement dimensionné pour éviter les pertes. Concrètement, la fenêtre temporelle d'identification est calculée de façon conservative : on garantit que même le paquet arrivant le plus rapidement sera reconnu avant que la fenêtre ne se ferme. C'est un choix de conception sûr (safe by design) typique de l'avionique.

UTC — Unité de Traitement des Communications

L'UTC est le bloc central de l'IOM. Il joue deux rôles complémentaires selon le sens du flux de données :

Réceptionne les paquets 429 venant de l'extérieur de l'étagère et les ré-émet vers les modules cœurs via le réseau AFDX. L'UTC encapsule les labels 429 dans des trames AFDX.

Récupère les trames AFDX venant des modules cœurs via le réseau AFDX, et génère les paquets 429 correspondants pour envoi vers les équipements extérieurs de l'étagère.

Le module GWM — Gateway Module

Définition : Le GWM (Gateway Module) est la passerelle entre l'étagère IMA et un réseau Ethernet standard non avionique. Il partage la même structure de base que l'IOM (UTC + UER AFDX), mais substitue les unités ARINC 429 par une UER Ethernet pour s'interfacer avec le monde extérieur.

Analogie : si l'IOM est le traducteur ARINC 429 ↔ AFDX, le GWM est son équivalent pour le protocole Ethernet standard ↔ AFDX. Il sert notamment à connecter des systèmes de maintenance, des outils de chargement logiciel ou des équipements de cabine qui s'expriment en Ethernet mais doivent communiquer avec le cœur avionique certifié.

Architecture interne et comparaison avec l'IOM

UER Ethernet — spécifique au GWM

En entrée : filtre les trames Ethernet selon les adresses et récupère uniquement celles à traiter.

En sortie : émet les données vers le réseau Ethernet après consultation du contenu de la trame. Spécifique GWM
UTC + UER AFDX — commun à tous les modules

Fonctionnement identique à celui de l'IOM : l'UTC gère la traduction bidirectionnelle, l'UER AFDX assure la connexion au réseau de l'étagère. Idem IOM & CPIOM
Cohérence architecturale : Le principe de conception est uniforme entre CPIOM, IOM et GWM — tous partagent un bloc UER AFDX identique en bas de leur pile fonctionnelle. Seule la couche d'interface extérieure varie (processeur applicatif pour CPIOM, ARINC 429 pour IOM, Ethernet pour GWM). Cette homogénéité facilite la maintenance, la reconfiguration et la certification en limitant le nombre de composants distincts à qualifier.

L'ADCN — Aircraft Data Communication Network

Définition : L'ADCN (Aircraft Data Communication Network) est le réseau de communication global de l'avion. Il fédère l'ensemble des échanges entre toutes les étagères IMA, les équipements capteurs/actionneurs et les LRU, en s'appuyant sur un réseau cœur AFDX et plusieurs réseaux périphériques de technologies hétérogènes.

Structure du réseau — une fédération de bus

L'ADCN n'est pas un réseau homogène : c'est une fédération de technologies complémentaires. Chaque type de bus répond à des contraintes spécifiques de débit, de déterminisme, de certification et de compatibilité avec l'existant.

AFDX Réseau cœur (backbone)
CAN Systèmes mécaniques
ARINC 429 Avionique legacy
Analogique Capteurs directs
Discret Signaux binaires

L'AFDX joue le rôle de colonne vertébrale : les switches AFDX assurent la connectivité entre toutes les étagères IMA et les modules CPM/CPIOM. Les IOM font le pont vers les bus ARINC 429 périphériques, les GWM vers Ethernet.

Organisation de l'A380 : Le réseau ADCN de l'A380 est organisé en domaines fonctionnels (Flight Control, Cockpit, Engines, Fuel & Landing Gear, Cabin). Chaque domaine regroupe ses CPM, IOM et LRU autour d'une paire de switches AFDX redondants (S1 à S4). Cette organisation en domaines permet une isolation des pannes et une certification modulaire par domaine plutôt que du système entier.

Le monde ouvert et le SCI

L'ADCN n'est pas hermétiquement isolé du reste du monde. Airbus et les compagnies aériennes ont besoin d'accéder au réseau avionique depuis des systèmes non certifiés : gestion de maintenance, téléchargement de logiciels via le protocole A615-A (TFTP), gestion centralisée des BITE (Built-In Test Equipment). Ces usages constituent le monde ouvert.

SCI — Secure Communication Interface : Passerelle sécurisée qui constitue l'unique point d'interconnexion autorisé entre le monde ouvert (réseau Ethernet airline / Airbus) et le monde avionique (réseau AFDX certifié). Toute communication entre les deux mondes transite obligatoirement par le SCI.

Fonctionnalités du SCI

  • Passerelle AFDX / Ethernet — traduit les trames entre les deux protocoles réseau sans exposer directement l'un à l'autre
  • Translation d'adresses IP (NAT avionique) — les adresses IP du monde avionique sont masquées du monde ouvert ; le SCI les remplace par des adresses translatées définies dans une table de configuration statique et certifiée
Séparation impérative — safety et security : Le monde ouvert et le monde avionique certifié ne doivent jamais être directement interconnectés. La raison est double : d'une part la safety (une erreur de configuration dans le réseau de maintenance ne doit pas affecter les systèmes de vol), d'autre part la security (une intrusion dans le réseau airline ne doit pas permettre d'accéder aux systèmes critiques). Le SCI est la mise en œuvre matérielle et logicielle de cette frontière.

Projets futurs — Réutilisation de l'IMA

L'IMA développée pour l'A380 est une architecture mature et maîtrisée. Plutôt que de repartir de zéro à chaque nouveau programme, Airbus a choisi de la réutiliser et de l'adapter — une décision guidée par la réduction des risques et la disponibilité immédiate d'une base certifiée.

Architecture mature et disponible immédiatement. Maîtrise du système acquise sur A380 ⇒ risques de développement minimaux. Base de certification déjà constituée.

A400M (transport militaire), A350 (long-courrier), A320 NEO (New Engine Option — moyen-courrier).

IMA2G — Seconde Génération

IMA2G : Programme d'évolution vers une architecture de seconde génération, développé dans le cadre du projet européen SCARLETT (40 partenaires industriels et académiques, horizon 2015). Objectif principal : gagner en flexibilité et élargir les domaines d'application au monde ouvert, tout en capitalisant sur les acquis de l'IMA première génération.

Évolutions matérielles — vers les DME

  • DME (Distributed Modular Electronics) — les CPIOM sont remplacés par des modules électroniques distribués physiquement dans l'avion, plus proches des équipements qu'ils pilotent
  • Séparation E/S et calcul — les entrées/sorties et les composants de calcul sont découplés physiquement, permettant une évolution indépendante des deux
  • Calculateurs multi-cœurs — introduction de processeurs à plusieurs cœurs pour augmenter la puissance de calcul sans multiplier les boîtiers

Évolutions logicielles — vers la certification incrémentale

  • Indépendance plateforme / applications — une application certifiée peut migrer sur un nouveau matériel sans déclencher une re-certification complète du système
  • Certification incrémentale — on certifie uniquement les modifications, pas la totalité du système à chaque évolution logicielle ou matérielle
  • Chaîne d'outils SCARLETT — fournit un environnement intégré couvrant le développement, l'intégration, la certification et la maintenance
  • Standardisation communautaire — maximum de composants et d'outils normalisés pour maximiser la réutilisation entre acteurs industriels

Mécanismes de reconfiguration dynamique

L'une des ruptures majeures d'IMA2G : la gestion de la redondance, l'allocation des ressources et leur affectation aux applications sont pilotées dynamiquement par la plateforme, au lieu d'être figées statiquement lors de la conception du système.

Analogie statique vs dynamique : dans l'IMA classique, chaque application se voit attribuer une tranche de CPU et une plage mémoire fixes pour toute la durée de vie du système — comme des postes de travail dédiés et inamovibles. Dans IMA2G, la plateforme peut redistribuer ces ressources à la volée — comme un gestionnaire qui réaffecte les places selon l'occupation réelle et les pannes éventuelles. En cas de défaillance d'un module, la plateforme reconfigure automatiquement l'allocation sur les modules valides restants.

PikeOS SSV — Safe and Secure Virtualisation

PikeOS SSV : Hyperviseur temps réel certifiable sur lequel s'appuie l'architecture SCARLETT. Basé sur un micro-noyau, il fournit une interface logicielle pour les machines virtuelles tout en garantissant le partitionnement spatial et temporel entre partitions. La para-virtualisation rend les applications indépendantes du matériel physique sous-jacent.
62 Partitions max
< 10 000 Lignes de code (noyau)
Para-VM Indépendant du matériel
  • Micro-noyau — base logicielle minimale : surface d'attaque réduite, analyse de code facilitée, certification peu coûteuse
  • Partitions totalement indépendantes — chaque partition peut démarrer et s'arrêter indépendamment sans affecter les autres
  • Para-virtualisation — les applications voient une machine virtuelle abstraite, indépendante du matériel physique sous-jacent ; un portage sur nouveau hardware ne nécessite pas de modifier l'application
  • < 10 000 lignes de code — base de certification très réduite par rapport à un OS complet → coût de certification drastiquement abaissé
Tendance de fond : Le graphe d'évolution des avions Airbus (Concorde → A380 → futur) illustre le paradoxe de l'IMA : à mesure que la fonctionnalité avionique progresse exponentiellement (échelle logarithmique), le nombre d'équipements électroniques physiques s'effondre grâce à la consolidation IMA. IMA2G pousse cette logique encore plus loin : toujours plus de puissance logicielle et de flexibilité, toujours moins de boîtiers physiques à câbler, certifier et maintenir.
