IMA — Les modules (CPIOM)

Synthèse

  • CPIOM — module calculateur IMA d'Airbus (A380), hébergeant jusqu'à 16 partitions logicielles sur un hardware partagé (CPU MPC755 + board IO_DSP)
  • ACR — Avionics Computing Resource (DO255) : plateforme générique standardisée fournissant isolation mémoire (MMU), créneaux CPU fixes et communications séparées par partition
  • Partitionnement — double isolation spatiale (MMU, débordement impossible) et temporelle (tranches fixes sans préemption) ; toute violation est détectée, bloquée et reportée sans impact sur les autres partitions
  • APEX / ARINC 653 — OS standardisé à deux niveaux : ordonnancement statique des partitions (CPM), ordonnancement dynamique préemptif des processus au sein de chaque partition
  • Certification — séparation ACR Platform Provider / Application Providers / Integrator, chacun certifié DO178B indépendamment → TC/STC auprès FAA/JAA

Le CPIOM dans l'architecture IMA Airbus

Dans l'IMA Airbus (A380), le principe fondamental est le partage de ressources : plusieurs fonctions avioniques cohabitent sur le même calculateur physique. Les applications sont regroupées en partitions qui se concurrencent pour l'accès à deux familles de ressources communes.

Temps processeur et mémoire RAM — alloués de façon statique et déterministe entre partitions selon une table figée à la conception.

Accès réseau AFDX / ARINC 429 — chaque partition dispose de son propre créneau d'accès, géré par l'OS ARINC 653.

CPIOM — Core Processing and Input/Output Module : module IMA d'Airbus implémentant la plateforme ACR. Il héberge jusqu'à 16 partitions logicielles concurrentes sur un seul calculateur physique, tout en garantissant leur isolation mutuelle complète via le partitionnement spatial et temporel.
Analogie — l'immeuble de bureaux : Le CPIOM fonctionne comme un immeuble dont chaque locataire (partition) occupe un appartement séparé avec ses propres clés (zone mémoire réservée) et ses horaires fixes (créneau CPU). L'immeuble (hardware) est commun, mais les murs porteurs (MMU) empêchent toute intrusion entre appartements. Si un locataire provoque un incendie (bug applicatif), les coupe-feux (mécanismes anti-propagation) protègent les voisins.

Architecture logicielle : la pile CPIOM

Le CPIOM distingue deux grandes catégories de logiciels :

Resident Software Non rechargeable

Logiciel embarqué en mémoire non volatile, intimement lié au hardware. Ne peut pas être mis à jour par téléchargement en exploitation. Exemples : firmware de démarrage bas niveau, code lié au FPGA de partitionnement.
Non Resident Software Téléchargeable

Logiciel pouvant être mis à jour via le service DLCS (Data Loading and Configuration System). Comprend les Drivers (couche Hardware Abstraction) et l'Executive Software / OS (APEX, ARINC 653).

La pile logicielle complète s'organise en couches empilées, séparées par des interfaces standardisées :

Interface COEX (COre / EXecutive Software Interface) — point de découplage entre l'OS (Executive Software) et les pilotes hardware (Driver Software). Grâce à COEX, l'Executive Software peut être porté sur différents boards matériels sans modifier les applications. C'est l'équivalent d'une Hardware Abstraction Layer dans un OS classique.

Architecture matérielle

Le CPIOM se compose de deux boards physiques distinctes, communicant via un bus USB interne (Intra LRM Communication) :

CPU Board Calcul principal

  • Processeur CPU MPC755 (PowerPC 32 bits)
  • RAM, Flash, NVM (connectés via bus PCI)
  • Partitioning Control (MMU matérielle)
  • Interface réseau AFDX (bus avionique haute vitesse)
  • Ports DSI/DSO (entrées/sorties discrètes)
IO_DSP Board I/O temps réel

  • DSP TMS320C33 (Texas Instruments) + RAM interne
  • Interface AUDIO (communications vocales)
  • Flash, DSI/DSO, DGI (entrées discrètes génériques)
  • Piloté par IO SW, séparé du Basic Software

Le Basic Software est la couche système commune aux deux boards. Il regroupe :

  • A653 Operating System + Drivers + Boot — noyau ARINC 653 certifié
  • UTIL : NVM Management, Module Management
  • ResourceBITE — auto-test intégré des ressources hardware au démarrage
  • SNMP-MIB — gestion réseau (maintenance et supervision)
  • Dataload A615A — service de téléchargement conforme au standard A615A
  • Instrumentation — monitoring des performances et collecte des métriques

L'ACR : plateforme générique certifiable

ACR — Avionics Computing Resource (standard DO255) : plateforme embarquée générique fournissant un environnement d'exécution standardisé, indépendant des applications hébergées.

"Un ACR correct est vu comme une ressource standardisée et ré-utilisable pour les applications avioniques."

L'ACR apporte trois bénéfices fondamentaux qui distinguent l'IMA de l'architecture fédérée :

Les applications sont développées contre l'API ARINC 653 — elles ignorent le hardware sous-jacent. Un changement de board (ex. : nouveau processeur plus rapide) ne nécessite pas de modifier l'applicatif.

Des applications de niveaux DAL différents (ex. DAL-A pour le contrôle de vol, DAL-D pour l'info passagers) cohabitent sur le même calculateur, certifiées indépendamment.

L'ACR est configurable par tables statiques : applications hébergées, catégories d'avions (A318/A320/A380…). La même plateforme physique peut équiper toute la famille Airbus.

Les services fournis par l'ACR aux applications (définis par DO255) :

  • API standardisée ARINC 653 — interface commune à tous les calculateurs IMA, indépendante du hardware
  • Isolation et protection des applications — chaque application est confinée dans sa partition
  • Détection des violations de partition — toute tentative est bloquée et journalisée
  • Communication avec le reste de l'avion — via AFDX, ARINC 429, ou autres bus
  • Ressources d'exécution garanties — CPU, mémoire, I/O alloués contractuellement
  • Téléchargement — mise à jour du logiciel en service via DLCS
  • Monitoring — surveillance permanente de l'état de fonctionnement
Le principe API de l'ACR (DO255) : L'API expose 7 domaines de services : management des partitions · management des applications (dans une partition) · management du temps · management de la mémoire · management des communications · monitoring · point de téléchargement.

Partitionnement : isolation temporelle et spatiale

Le mécanisme de partitionnement est le cœur de l'ACR. Il garantit une double isolation — spatiale (mémoire) et temporelle (CPU) — entre toutes les partitions.

Isolation temporelle — partage statique du temps

Le processeur est découpé en tranches de temps fixes (time slots), définies une fois pour toutes à la conception du système :

  • Chaque partition reçoit son segment périodique d'exécution garanti
  • Pas de notion de priorité ni de préemption entre partitions — seul le timer change le contexte
  • Si une partition n'utilise pas son créneau, le temps restant est perdu (jamais redistribué)
  • Tout dépassement de créneau est une violation temporelle, immédiatement sanctionnée
Attention — le temps non utilisé est perdu : Une partition qui termine son traitement avant la fin de son créneau ne "rend" pas le temps restant. Ce comportement est intentionnel : il garantit le déterminisme. Aucune partition ne peut accaparer les ressources d'une autre, même si celle-ci est inactive.

Isolation spatiale — partage statique de la mémoire

La mémoire est partitionnée une fois pour toutes lors de la configuration système :

  • Chaque partition dispose d'une zone mémoire exclusive définie dans une table d'allocation statique
  • La MMU (Memory Management Unit) intercepte chaque accès mémoire et vérifie qu'il appartient à la zone autorisée pour la partition courante
  • Tout débordement (buffer overflow) est physiquement impossible — la MMU bloque l'accès avant qu'il n'atteigne la mémoire cible
  • La table d'allocation est statique : elle ne change pas en cours d'exécution
Analogie — le vigile de la MMU : Avant qu'une instruction puisse lire ou écrire à une adresse mémoire, la MMU consulte la table d'allocation : "cette adresse appartient-elle bien à la partition courante ?" Si non, l'accès est refusé immédiatement et l'incident est signalé — comme un vigile qui refuse l'entrée d'un appartement à quelqu'un dont la clé ne correspond pas.

Mécanismes anti-violation

Principe fondamental (DO178B) : "Le partitionnement fournit une isolation entre des fonctions indépendantes afin d'isoler les fautes et de diminuer les coûts de certification."

Une erreur propagée (qui se diffuse à d'autres fonctions) est pire que de perdre la fonction elle-même. La ségrégation stricte empêche cette propagation.

L'ACR garantit que toute tentative de violation — temporelle ou spatiale — est traitée sans impacter les autres partitions :

  • Détection — interceptée matériellement (MMU pour l'espace, timer pour le temps)
  • Blocage — l'accès illicite est immédiatement bloqué, sans exécution
  • Report — l'incident est journalisé en NVM et remontable au monitoring
  • Isolation — aucune conséquence sur les autres partitions actives

Le modèle APEX : partitions et applications

APEX — APplication EXecutive, standardisé par ARINC 653 : OS du CPIOM définissant un modèle à deux niveaux d'ordonnancement imbriqués. Le niveau supérieur (partitions) est statique et certifiable formellement. Le niveau inférieur (processus) est dynamique et flexible.
Niveau système — partitions Ordonnancement statique

Un système = {partitions}. Chaque partition regroupe des fonctions avioniques. L'ordonnancement des partitions par le CPM est figé à la conception — pas de décision dynamique. La partition est l'unité de ségrégation spatiale ET temporelle.
Niveau partition — applications Ordonnancement dynamique

Une partition = {applications/processus}. Au sein d'une partition, les processus sont ordonnancés de façon dynamique et préemptive avec priorités. Les défaillances d'un processus restent confinées dans la partition.
Pourquoi deux niveaux d'ordonnancement ? Le niveau supérieur (partitions) doit être certifiable — une table statique est vérifiable formellement et son comportement est prouvable. Le niveau inférieur (processus au sein d'une partition) peut être plus souple car toute défaillance reste cloisonnée. Ce découplage est un compromis optimal entre rigueur de certification et flexibilité de développement applicatif.

Objectifs du management des applications (ARINC 653) :

Conception, tests et vérification de chaque processus s'effectuent indépendamment. Un équipementier peut certifier son application sans connaître les autres.

Un dysfonctionnement d'un processus reste confiné à sa partition. Il ne peut pas corrompre l'état d'une autre application ou partition.

L'ajout de nouvelles applications n'a aucune conséquence sur les applications déjà présentes. On peut certifier les nouvelles applications séparément.

Services ACR détaillés

Management du temps

L'ACR gère l'exécution des partitions ET des processus (dans chaque partition) pour maintenir un comportement déterministe à l'échelle de la plateforme entière. La règle fondamentale : l'altération des paramètres temporels d'une application dans une partition ne doit avoir aucune conséquence sur les autres partitions.

Niveau plateforme
Ordonnancement statique des partitions — périodes fixes, déterministes
Niveau partition
Ordonnancement dynamique préemptif des processus — priorités configurables
Isolation
Un bug de timing dans une partition n'affecte pas le scheduling des autres

Management des communications

Les communications sont également allouées statiquement à la conception. Chaque partition a son propre créneau d'accès réseau. L'application passe par l'OS ARINC 653 qui dispatche vers le bus physique adéquat (AFDX, ARINC 429…) :

Exemple d'appel APEX : Write_Sampling_Port(portId, data, length)
L'application écrit dans un port logique. L'OS consulte la Table de Communications Statique pour savoir sur quel bus physique router le message. L'application ne connaît ni le bus, ni la topologie réseau.
Sampling vs Queuing ports (ARINC 653) : Les Sampling Ports écrasent le dernier message (modèle "dernière valeur connue" — utilisé pour des données continues comme cap, vitesse, altitude). Les Queuing Ports maintiennent une file FIFO (utilisé pour des événements discrets comme des commandes ou des alarmes).

Monitoring

L'ACR surveille en permanence l'état de toutes les partitions. Les rapports de monitoring sont stockés en mémoire non volatile (NVM) pour extraction ultérieure et contiennent :

  • Erreurs et exceptions relevées par les applications
  • BIT status (Built-In Test Equipment) — résultats de l'auto-test au démarrage
  • Utilisation maximale des ressources (CPU, mémoire, communications) par partition
  • Tentatives de violation détectées et bloquées
  • Autres données de maintenance
Pourquoi stocker en NVM ? En cas de panne ou de redémarrage, les données de monitoring doivent être préservées pour permettre le diagnostic post-incident. La NVM garantit la persistance même en cas de coupure d'alimentation complète.

Téléchargement

L'ACR fournit le service de téléchargement via le DLCS (Data Loading and Configuration System), conforme au standard A615A. Ce service permet la mise à jour du logiciel embarqué en service sans démontage du module.

  • Réception et intégrité — vérification que les données téléchargées sont complètes et non corrompues
  • Confirmation au DLCS — acquittement de bonne réception envoyé au Ground Support Equipment
  • Identification des versions — enregistrement du Part Number et Software Version loaded pour traçabilité

Processus d'allocation et de certification

Processus d'allocation des ressources

Avant tout développement, les ressources de chaque application doivent être déclarées et allouées contractuellement. Ce processus en trois temps est garant du déterminisme global :

  • Déclaration des besoins — chaque Application Provider déclare ses besoins en CPU, mémoire et bande passante communications
  • Concertation et allocation — le Platform Integrator consolide les besoins, vérifie la faisabilité et alloue formellement les ressources
  • Développement contraint — l'application est développée et testée dans le strict respect des ressources allouées (le dépassement en run-time déclenche une violation)
Conséquence de l'allocation contractuelle : Une application qui dépasserait ses ressources allouées à l'exécution est détectée et sanctionnée (violation). Il n'y a pas de "tolérance" — les garanties sont enforced matériellement. C'est le prix du déterminisme certifiable.

Indépendance des partitions et bénéfice DAL

Le partitionnement permet une avancée majeure pour la certification :

Des systèmes avec des niveaux de DAL différents (DAL-A, DAL-C…) peuvent être développés sur le même calculateur, chacun certifié indépendamment. La modification d'une application n'affecte pas la certification des autres.

Temps processeur, mémoire et communications réseau sont définis par tables statiques configurables. La même plateforme physique s'adapte à différentes configurations avion (A318, A320, A380…).

Bénéfice clé du partitionnement pour la certification : Grâce à la ségrégation, chaque Application Provider certifie son logiciel de façon indépendante, sans connaître les autres applications hébergées sur le même CPIOM. Le coût total de certification est réduit, car on n'est pas obligé de qualifier l'ensemble de la plateforme au niveau DAL le plus strict hébergé.
